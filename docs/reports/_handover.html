3. إدارة الأسرار (Secret Management)

مبادئ الأمان:

  • الملف .env: ملف محلي غير ملتزم لتخزين متغيرات البيئة الحساسة أثناء التطوير
  • .env.example: نموذج آمن بدون قيم حقيقية — في git لأغراض التوثيق
  • .gitignore: يحظر جميع ملفات .env (سطر 13–14) — لا تُرتكب الأسرار مطلقًا
  • Coolify (الإنتاج): جميع الأسرار (DB_PASSWORD, SETTINGS_ENCRYPTION_KEY, ANTHROPIC_API_KEY, S3_*, BACKUP_ENCRYPTION_KEY, STRIPE_*) يتم إدارتها عبر واجهة Coolify "Environment Variables" — أبدًا في الملفات
  • الأسرار الحساسة خاصة: DB_PASSWORD, BOOTSTRAP_TOKEN, SETTINGS_ENCRYPTION_KEY, ANTHROPIC_API_KEY, S3_SECRET_KEY, BACKUP_ENCRYPTION_KEY, STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, P6_PASSWORD, AUTODESK_CLIENT_SECRET

إعدادات القيود:

  • لا تحتوي على علامات دولار ($) في القيم الحساسة — تفسدها استيفاء متغيرات Docker
  • استخدم أحرف أبجدية رقمية فقط للأسرار المشتركة (مثل DEMO_SEED_PASSWORD)
  • للأسرار الجديدة: استخدم openssl rand -hex 32 (مثال: BACKUP_ENCRYPTION_KEY)

4. التشغيل المحلي من الصفر (Local Setup from Scratch)

المتطلبات المسبقة:

  • Node 22+ (من npm، أو nvm)
  • MySQL 8 أو MariaDB 10.6+
  • Docker Desktop (للخيار العازل المسهل)

خطوة 1: استنساخ المستودع

  • git clone <repo-url> sigma-pmo
  • cd sigma-pmo

خطوة 2: إعداد قاعدة البيانات (الخيار أ: Docker — موصى به)

  • docker-compose up -d (يشغل MySQL 8 على منفذ 3306، مع حجم الاستمرار sigmadb، مع كلمة المرور الافتراضية rootpw)
  • انتظر 30 ثانية حتى يبدأ MySQL الفحص الصحي

خطوة 2 (الخيار ب: MySQL محلي)

  • تثبيت MySQL Server 8
  • إنشاء قاعدة بيانات: mysql -u root -p -e "CREATE DATABASE sigma_pmo"

خطوة 3: إعداد Backend

  • cd backend
  • cp .env.example .env
  • تحرير .env:
  • NODE_ENV=development
  • DB_HOST=localhost (أو db إذا كان Docker)
  • DB_USERNAME=root
  • DB_PASSWORD=rootpw (أو ما عيّنته)
  • DB_DATABASE=sigma_pmo
  • npm ci (تثبيت الاعتماديات المقفلة)
  • npm run build (بناء البرنامج)
  • npm run gen:samples (إنشاء بيانات نموذجية في ../data/samples/)
  • npm run migration:run (تطبيق الهجرات)
  • npm run start:dev (بدء API على http://localhost:3001/api/v1 في وضع المراقبة)

خطوة 4: إعداد Frontend (محطة طرفية منفصلة)

  • cd frontend
  • npm ci
  • npm run dev (بدء UI على http://localhost:3000)

خطوة 5: اختبار الدخان من طرف إلى طرف

  • cd backend
  • npm run ingest -- ../data/samples/schedule.xlsx (ابتلاع ملف نموذجي)
  • curl -s -X POST -H "Content-Type: application/json" \ -d '{"projectKey":"P-1000"}' \ http://localhost:3001/api/v1/rules/evaluate (تقييم القواعد)
  • curl -s -X POST -H "Content-Type: application/json" \ -d '{"projectKey":"P-1000","periodDays":7}' \ http://localhost:3001/api/v1/summary/generate (إنشاء ملخص)

سيناريوهات استكشاف الأخطاء:

  • فشل الاتصال بـ MySQL: تحقق من DB_HOST، DB_PORT، DB_PASSWORD؛ تحقق من صحة MySQL: mysql -h <host> -u root -p
  • خطأ الهجرة: تأكد من تطابق المخطط مع الهجرات؛ اختبر npm run migration:run يدويًا
  • CORS مرفوض: تعديل CORS_ORIGINS في .env (مثال: http://localhost:3000)
  • بدون مفتاح API Anthropic: النظام يعمل في الوضع الحتمي فقط — تخطي جميع استدعاءات Claude

ملاحظات مهمة:

  • الوضع الافتراضي SEED_DEMO=true، DEMO_LOGIN_PUBLIC=true — حسابات تجريب فورية متاحة (demo/SigmaDemo2026)
  • للبيانات الحقيقية: عيّن SEED_DEMO=false، DEMO_LOGIN_PUBLIC=false، استخدم npm run user:create لإنشاء حسابات حقيقية
  • جميع الملفات المرفوعة مخزنة في data/storage/ محليًا (أو S3 إذا تم تكوينها)
  • النسخ الاحتياطية الليلية الآلية تعمل عبر BackupService (Cron) — تتطلب S3 + BACKUP_ENCRYPTION_KEY للإنتاج

2 · بنية النظام (Architecture)

بنية النظام

Frontend (الواجهة الأمامية)

  • Next.js 16.2.6 + React 19، App Router (مجلد app/)
  • Standalone output mode — خادم Node.js مستقل على المنفذ 3000
  • مكتوبة بـ TypeScript + Tailwind CSS (Font: Tajawal للعربية)
  • Deployment: Coolify أو Vercel (صفحات الهبوط على Vercel)
  • متغيرات البناء (Build-time): NEXT_PUBLIC_API_BASE، NEXT_PUBLIC_DEMO_LOGIN

Backend (الخادم الخلفي)

  • NestJS 11 + TypeORM 0.3 + TypeScript
  • بادئة المسار الموحدة: /api/v1 (متحكم عام عبر app.setGlobalPrefix('api/v1'))
  • المنفذ: 3001 (داخل الحاوية)، مرتبط من خلال Traefik (Coolify)
  • Swagger/OpenAPI: /api/v1/docs (يعرض كل الـ routes تلقائياً)
  • الحجم الأقصى للطلب: 25 ميجابايت (قابل للتخصيص عبر bodyLimit)
  • التوثيق: رأس x-api-key (SHA-256 lookup)، يدعم x-bootstrap-token لتجاوز البدء

قاعدة البيانات (Database)

  • MySQL 8.4 (صورة رسمية)
  • TypeORM + Migrations:
    • يُشغَّل تلقائياً على البوت في الإنتاج (migrationsRun: isProd)
    • الملفات المترجمة: dist/src/migrations/*.js
    • في التطوير: يدار يدوياً عبر npm run migration:run
  • الترميز: UTF8MB4 (دعم العربية الكامل)
  • المنطقة الزمنية: UTC صراحة (Z)، العملاء يحولون عند العرض
  • ملف التخزين المستمر: sigmadb (volume مسمى)
  • Healthcheck: mysqladmin ping كل 10 ثوانٍ

الملفات الخارجية (External File Storage)

  • StorageService (في ingestion module) — تخزين مضمون بـ SHA-256
  • الوسيط: Cloudflare R2 أو S3-compatible عبر @aws-sdk/client-s3
  • البادئة (Prefix): sigma-system/ + شجرة مجزأة (XX/YY/HASH__filename)
  • الخيارات: S3_ENDPOINT, S3_REGION, S3_BUCKET, S3_ACCESS_KEY, S3_SECRET_KEY, S3_FORCE_PATH_STYLE

المهام والعمليات المجدولة (Jobs/Workers)

  • @nestjs/schedule + @Cron decorators
  • CommunicationsAlertsCron: كل ساعة — إنذارات 24 ساعة للاتصالات غير المقروءة، التصعيد
  • EvidenceCron: عملية الأدلة
  • BackupService: كل يوم في الساعة 2 صباحاً — نسخة احتياطية لـ MySQL إلى R2/S3 (مشفرة بـ AES-256-GCM)

AI Provider (مزود الذكاء الاصطناعي)

  • ClaudeService (modules/claude) — غلاف رقيق حول SDK الرسمي @anthropic-ai/sdk
  • النموذج: Anthropic Claude (Haiku/Sonnet/Opus)، مع تعيين modelTier إلى معرّفات النماذج
  • الميزات: Personas (مخازن مؤقتة ephemeral)، PolicyAddons (تعليمات inline)، ProjectMemory (حقائق مستفادة)
  • الاستخراج: مفتاح API من ANTHROPIC_API_KEY البيئة فقط (آمن)
  • الحالة: isEnabled() يرجع false عند عدم وجود مفتاح

عزل المستأجرين (Tenant Isolation)

  • TraceableEntity (base) — كل صف يحمل companyId مفهرسة
  • طلب المسار:
    • المستخدم → API (x-api-key)
    • ApiKeyGuard: يستخرج companyId من مستخدم مُفعَّل (أو مسموح Bootstrap)
    • TenantContextMiddleware: يفتح متجر AsyncLocalStorage بـ companyId
    • ProjectScopeGuard (بعد ApiKeyGuard): يتحقق من أن projectKey/projectId ينتمي إلى companyId الحالي
    • الخدمات: تستدعي companyScope() (يرجع {companyId} أو {})
    • TypeORM → MySQL: يرشح حسب companyId
    • JSON response → Frontend
  • الاستثناء: SUPER_ADMIN يقرأ عبر جميع الشركات

المساعد والبرامج الإضافية للسياسات (Personas & Policy Add-ons)

  • PersonasService — يحفظ والشخصيات (slug، إصدار، موثر بـ cache)، نموذج الافتراضي (Sonnet)
  • PolicyAddonsService — تعليمات مؤلفة مستخدم (موجهة إلى السطح: planning, engineering, governance, reports)
  • BlockStructure: نظام Persona + Custom addon = أساس موحد للذكاء الاصطناعي

الـWorkflow الكامل خطوة بخطوة

يوضح الجدول التالي كل مرحلة من مراحل معالجة البيانات والحوكمة في Sigma PMO، من استقبال القالب حتى مستوى الإدارة التنفيذية.

الخطوة الـ API المسؤول الجداول المتأثرة المدخلات المخرجات الربط بالمرحلة التالية
1. Template (القالب) GET /ingestion/template — طلب تحميل الملف ملف Excel متعدد الأوراق (Projects, Activities, Resources, Assignments, Reports) يملأ المستخدم البيانات ويرفعها
2. Ingestion (الاستقبال) POST /ingestion/upload أو POST /ingestion/ingest-path SourceFile, IngestionRun filename, contentBase64 (أو مسار محلي) runId, sourceFileId, parser (CSV/Excel/P6/MSProject), status=PENDING ينتقل إلى التحقق الفوري
3. Validation (التحقق) ValidationService.validate() (داخلي خلال Ingestion) IngestionRun (تحديث status) RawDataset (projects, activities, resources, assignments, reports) ValidationReport: passed (bool), errorCount, warningCount, issues[] إن نجحت → تطبيع، وإلا → رفض (FAILED)
4. Stored Records (السجلات المخزنة) NormalizerService.normalize() (داخلي في transaction) Project, Activity, Resource, ResourceAssignment, Report, ConfidenceScore Validated dataset counts: {project, activity, resource, assignment, report}, confidence scores (completeness, consistency, sourceReliability, overall) المسجلات الحالية جاهزة لتقييم القواعد
5. Rules (القواعد) - L1 POST /rules/evaluate RuleEvaluation, Alert projectKey أو projectId (اختياري = كل المشاريع) RuleEvaluationOutcome: evaluationId, alertCount, byCode{}, bySeverity{} كل alert لديه code, severity, summary، ومؤشرات على rows المؤثرة
6. Alerts (التنبيهات) - L2 GET /rules/alerts Alert (read-only للعرض) queries: evaluationId, projectId, projectKey, severity, limit قائمة alerts مع projectBusinessKey، activityId، resourceId، assignmentId، reportId، context (JSON) كل alert يدخل السياق القانوني/الحوكمة
7. Decisions (القرارات) - L3 POST /governance/decide أو POST /rules/workflows/run GovernanceDecision, GovernancePolicy ruleEvaluationId, projectKey (للقرار الحكومي) DecisionOutcome: policyId, policyVersion, decisionCount, byParty{}, byLevel{}. كل decision: responsibleParty، escalationLevel، fidicClause، rationale، interventions[] القرار محدد الطرف المسؤول والمستوى والعمل المقترح
8. Evidence (الأدلة) - Dispute GET /governance/alerts/:id/evidence EvidenceRoom, EvidenceFile, EvidenceItem, EvidenceChunk alertId EvidencePackage: chain كامل من alert → decision → rule source → confidence، مع ملفات مرفقة (PDF، صور)، hash تحقق أدلة معتمة قابلة للتحقق (integrity chain)
9. Approval (الموافقة) - Decision Review POST /governance/decisions/:id/review DecisionReview (append-only) decisionId، action (approve/reject/acknowledge)، comment، performer (User) ReviewRecordResult: reviewId، chainState (pending/awaiting-second-approval/approved/rejected)، requiresDualApproval (critical decisions) حالة القرار محدثة (pending → approved)
10. Audit (التدقيق) - L4 GET /governance/audit DecisionReview (joined with Decision, Alert) limit (default 100، max 500) قائمة audit rows: reviewId، createdAt، action، actorUserId، actorDisplay، decisionId، alertCode، severity، escalationLevel Trail سجل شامل → عرض L7 التنفيذي
11. Executive (التنفيذي) - L7 GET /executive/overview, GET /executive/kpis ExecutiveSummary، AnalyticsSnapshot، Project (read-only) projectKey (للتفاصيل) أو no param (محفظة) ExecutivePack: KPIs (health, confidence, forecast risk)، StrategicKpis (alignment, benefits realization, governance score)، PortfolioKpis (roll-up across projects) مؤشرات موحدة للإدارة والقرار
12. Governance-Command (الحوكمة) - L8 POST /governance-command/recompute, GET /governance-command/overview CorrectiveAction، GovernanceStatusSnapshot، Project (current) nodeKey, nodeType (PROJECT/PROGRAM/PORTFOLIO) ConsolidatedNode: governanceStatus (green/yellow/orange/red), openActions[], escalationPaths[], recommendedActions[], impactAnalysis (VAR + benefit realization) استراتيجية شاملة وتوصيات تنفيذية

ملخص الربط والتدفق

  • Template → Ingestion: يحمل المستخدم ملف Excel المعرّف من API ويرفعه عبر POST /ingestion/upload
  • Ingestion → Validation → Stored Records: داخلياً في IngestionService.ingest(): parse → validate → normalize في transaction واحد. إن فشل التحقق، status=FAILED. إن نجح، السجلات تُكتب (Project, Activity, وغيرها) و status=NORMALIZED مع ConfidenceScore
  • Stored Records → Rules (L1): POST /rules/evaluate ينسخ الحالة الحالية (RuleEvaluation) ويطبق 10 قواعد (SCHEDULE_FINISH_SLIPPED، COST_OVERRUN، إلخ) مقابل snapshot. كل match → Alert
  • Rules → Alerts (L2): Alerts قابلة للاستعلام عبر GET /rules/alerts مع filters (evaluationId، projectKey، severity). كل alert نقطة فعل
  • Alerts → Decisions (L3): POST /governance/decide يطبق GovernancePolicy (accountability matrix، FIDIC mappings، escalation tiers) على كل alert → GovernanceDecision (responsibleParty، escalationLevel، rationale)
  • Decisions → Evidence (Dispute Track): POST /evidence/rooms/:id/commit يخزن findings مع hash integrity chain. GET /governance/alerts/:id/evidence يسترجع الـ evidence للنزاعات
  • Decisions → Approval (Decision Review): POST /governance/decisions/:id/review يسجل approve/reject/acknowledge من المسؤولين. القرارات الحرجة تتطلب dual approval. GET /governance/decisions/:id/trace يظهر السلسلة الكاملة من decision إلى confidence
  • Approval → Audit: كل review action مسجل في DecisionReview (append-only). GET /governance/audit يعرض trail الكامل للموافقات والأفعال
  • Audit + Execution → Executive (L7): GET /executive/kpis يحسب deterministic headlines (health، forecast risk، confidence average). GET /executive/scores يحسب enterprise governance scorecard (6 scores: governance، investment، portfolio، opportunity، bankability، funding)
  • Executive → Governance-Command (L8): POST /governance-command/recompute يدمج KPIs + open actions + escalation paths في ConsolidatedNode لكل project/program/portfolio. يصدر status (green/yellow/orange/red) و recommended corrective actions + impact analysis (value-at-risk، benefit realization)
  • Agent Pipeline (L1→L8): POST /agents/pipeline/run يشغّل orchestrator عبر جميع layers في تسلسل واحد (يدوي أو مجدول). كل agent لديه AgentExecution audit trail، model-tier config، health metrics

نقاط رئيسية

  • Traceability: كل Alert مرتبط بـ SourceFile (hash SHA-256) و IngestionRun و RuleEvaluation. GET /governance/decisions/:id/trace يسحب السلسلة الكاملة: decision → alert → rule → ingestion run → source file
  • Multi-Tenancy: جميع الـ APIs تحترم companyId (company-scoped). Null companyId = super-admin (رؤية الكل)
  • Append-Only Audit: DecisionReview و AuditLog و AgentExecution كلها append-only. لا تحديثات destructive
  • Governance Policy Versioning: كل قرار يدرج policyId و policyVersion. التحديثات تنشئ نسخة جديدة، لا تحدث الحالية
  • Confidence Scoring: ConfidenceScore محسوبة في Ingestion time (completeness، consistency، sourceReliability، overall). تُستخدم في Executive KPIs والـ Evidence chain
  • Dual Approval for Critical: القرارات من alerts ذات severity=critical تتطلب dual approval. chainState يتتبع حالة الموافقة (pending → awaiting-second-approval → approved)

8 · الأمن (Security)

الأمن (Security)

المصادقة (Authentication)

يعتمد النظام على مصادقة API key متعددة المستويات:

  • x-api-key Header: كل طلب API يتطلب رأس x-api-key يتم التحقق منه عبر SHA256 hash lookup في قاعدة البيانات (ملف api-key.guard.ts، السطر 57)
  • SHA256 Hashing: المفتاح الخام يتم تحويله إلى SHA256 قبل البحث في الجدول (auth.service.ts، السطر 31-32)
  • Multi-Session API Keys: كل مستخدم يحتفظ بـ 5 مفاتيح API نشطة كحد أقصى (MAX_SESSIONS = 5)، مما يسمح بجلسات متزامنة دون تسجيل الخروج (السطر 36 و61-69 في auth.service.ts)
  • Scrypt Password Hashing: كلمات المرور تُشفر باستخدام Scrypt مع ملح عشوائي 16-byte per-user:
    • SCRYPT_KEYLEN = 64
    • SCRYPT_COST = 16384 (N=2^14 محافظ)
    • Timing-safe comparison لمنع timing attacks (السطور 73-84)
  • Bootstrap Mode: في production يتطلب x-bootstrap-token header يطابق متغير BOOTSTRAP_TOKEN البيئي عند إنشاء أول admin (السطور 149-165 في auth.service.ts)

التفويض (Authorization)

يعتمد النظام على بنية قدرات (capabilities) معرّفة بدقة:

  • @RequiresCapability Decorator: كل endpoint محمي يطلب قدرة محددة، تُفرض عبر ApiKeyGuard (ملفات require-capability.decorator.ts وapi-key.guard.ts)
  • Role-Based Matrix: 13 دور مختلف (sigma_admin, sigma_reviewer, client, consultant, contractor, subcontractor, owner, operator, investor, lender, pmo, governance_board, bank, government_regulator, asset_manager) مع 50+ قدرة لكل دور في roles.enum.ts
  • Dynamic Overrides: CapabilitiesService يدمج القيم الافتراضية المرمزة مع جداول RoleCapabilityOverride في الذاكرة، مع حماية ضد القفل (canRead و canManageRoles لا يمكن تعطيلهما، sigma_admin غير قابل للتغيير)
  • Admin-Only Controls: القدرات الحساسة مثل canManageRoles وcanManagePlatform مقصورة على sigma_admin فقط

عزل المستأجرين (Tenant Isolation)

النظام يفرض عزل البيانات بين الشركات (tenants):

  • Tenant Context: AsyncLocalStorage يحتفظ بـ companyId الحالي per-request (ملف tenant-context.ts)
  • ApiKeyGuard Enforcement: يستخرج companyId من المستخدم المصرح وينشئه في tenantStorage (السطور 74-75)
  • ProjectScopeGuard (403): guard عام يتحقق من أن أي projectKey أو projectId في الطلب ينتمي إلى شركة المستخدم الحالية، ويرفع ForbiddenException (ملف project-scope.guard.ts، السطور 41-44)
  • Company Status Gates: المستخدمون من شركات معلقة أو ملغاة أو ذات فترات تجريبية منتهية يُرفضون برمز 403 (السطور 109-127 في auth.service.ts)
  • Scoped Data Queries: جميع خدمات البيانات تستخدم companyScope() لتصفية الاستعلامات

رؤوس الأمان (Security Headers)

تُطبق عبر helmet middleware في main.ts (السطور 43-48):

  • x-powered-by Disabled: app.disable('x-powered-by') يزيل رأس الكشف عن الخادم (السطر 24، audit 2026-06-28)
  • Helmet Configuration: تُطبق الرؤوس التالية افتراضياً:
    • Strict-Transport-Security (HSTS) - فرض HTTPS
    • X-Frame-Options - منع clickjacking
    • X-Content-Type-Options: nosniff
    • Referrer-Policy - التحكم في مرجع الطلب
    • Permissions-Policy - التحكم في ميزات المتصفح
  • CSP Disabled: Content-Security-Policy معطل لأن frontend هو origin منفصل

CORS (تقاسم الموارد بين المواقع)

يُتحكم فيه عبر متغير CORS_ORIGINS (main.ts، السطور 75-88):

  • في production: قائمة بيضاء من origins محددة عبر متغير البيئة (فاصلة منقوطة)
  • في development: يسمح localhost:3000 و 127.0.0.1:3000 و [::1]:3000
  • credentials: false (لا توصيلات ملفات تعريف)

تحديد معدل الطلب (Rate Limiting)

يُنفذ عبر @nestjs/throttler في throttler.module.ts:

  • Global Throttler Guard: مسجل كـ APP_GUARD (السطر 46)
  • Buckets:
    • default: 600 req/min (TTL 60s) - حماية عام
    • auth: 10_000_000 req/min globally (لكن @Throttle({ auth: { limit } }) على /auth/login تقلل إلى 10 req/min)
    • ingest: 10_000_000 req/min globally (لكن مسارات البث محدودة بـ 30 req/min)
    • ai: 10_000_000 req/min globally (لكن مسارات AI محدودة بالقيم الآمنة)
  • Dev Multiplier: في non-production تُضرب كل قيمة ب 50× لتجنب تحديد معدل خلال StrictMode و Fast Refresh
  • Configuration: throttlerDefaultLimit و throttlerDefaultTtlMs و throttlerAuthLimit و throttlerIngestLimit من البيئة

سجل التدقيق (Audit Logging)

يُنفذ عبر AuditInterceptor في audit.interceptor.ts:

  • Scope: يسجل جميع طلبات الـ mutation (POST, PUT, PATCH, DELETE) وجميع محاولات تسجيل الدخول (السطور 28-53)
  • Logged Fields: companyId, actorUserId, actorEmail, actorRole, action, method, path (512 char limit), statusCode, IP address (مع x-forwarded-for support)
  • Bodies Not Stored: لا يتم تخزين bodies الطلب - كلمات المرور و secrets لا تصل أبداً إلى السجل (السطر 35)
  • Failed Logins: محاولات تسجيل دخول فاشلة تُسجل مع attemptedEmail في meta (السطور 55-63)
  • Fire-and-Forget: الكتابة غير متزامنة ويتم بلعها على الخطأ بحيث لا تكسر أي طلب (السطور 94-97)
  • Governance Audit: GET /governance/audit يوفر سجل قرارات الحوكمة المشروط بـ tenant (السطور 146-178 في governance.controller.ts) - يربط قراءات القرار (approval/rejection/acknowledgement) برموز القرارات والتنبيهات
  • Audit Retrieval: GET /audit يسترجع entries محدودة (max 500) مع scoping تلقائي لـ tenant الحالي (audit.controller.ts)

إدارة كلمات المرور (Password Management)

  • Hashing Algorithm: Scrypt مع ملح عشوائي 16-byte per-user (auth.service.ts، السطور 73-76)
  • Parameters: SCRYPT_KEYLEN=64, SCRYPT_COST=16384
  • Verification: Timing-safe comparison لمنع timing attacks (السطور 79-84)
  • Demo Accounts: في UAT/production لا يمكن لحسابات demo المصرحة (isDemo=true) التوثيق إلا على صندوق demo حيث DEMO_LOGIN_PUBLIC=true (السطور 90-92)
  • Demo Password: كلمة مرور demo غير مميزة (مثل SigmaDemo2026) تُستخدم لحسابات demo العام - يُمكن تدويرها في البيئة
  • Admin Password: كلمة مرور admin البذرية من متغير ADMIN_SEED_PASSWORD؛ عند تركها فارغة يُنشئ النظام كلمة مرور عشوائية قوية ويسجلها مرة واحدة فقط (configuration.ts، السطور 161-162)

المراقبة (Observability)

البنية الأساسية للسجلات المهيكلة

يستخدم النظام مكتبة Pino JSON المهيكلة عبر NestJS لتسجيل جميع الطلبات والعمليات. في بيئة الإنتاج، يتم إصدار السجلات كـ ndjson (newline-delimited JSON) سطر واحد لكل عملية، بينما في التطوير يتم طباعتها بشكل مقروء. جميع السجلات مرتبطة تلقائياً بـ x-request-id الفريد.

معرّف الطلب وتتبع الارتباط

  • يتم توليد x-request-id في RequestIdMiddleware كـ UUID v4 أو يتم قبول معرّف موجود من رأس الطلب الوارد (للوكلاء العكسيين)
  • يتم إرجاع المعرّف في رأس الاستجابة x-request-id حتى يتمكن العملاء من الربط بين الطلب والاستجابة
  • تربط خيارات pinoHttp.genReqId تلقائياً هذا المعرّف في كل سطر سجل لمرة واحدة
  • مستويات السجل المخصصة: 500+ = خطأ، 400+ = تحذير، 200-399 = معلومات

واجهة حالة الوظائف الموحدة: GET /jobs

  • GET /jobs — قائمة جميع عمليات الإدخال (تصفية حسب الشركة) مع حالة موحدة
  • GET /jobs/:id — تفاصيل عملية محددة
  • الحالات الموحدة: queued (PENDING) | running (PARSED, VALIDATED) | completed (NORMALIZED) | failed (FAILED)
  • الملخص الإجمالي: إجمالي العدد، الصفوف حسب الحالة، failureRatePct = (عدد الفشل / المكتملة + الفشل) × 100
  • كل وظيفة تحتوي على: id، نوع، تسمية (اسم المحلل)، الحالة، الحالة الأولية، وقت البدء/الانتهاء، rowCounts، رسالة الخطأ (إن وجدت)

تسجيل فشل الإدخال والعمليات

  • IngestionRun — كل عملية إدخال لملف المصدر تحفظ status (PENDING, PARSED, VALIDATED, NORMALIZED, FAILED) و summary JSON تحتوي على رسالة الخطأ والإحصائيات
  • AgentExecution — كل تشغيل وكيل (L0–L8 أو امتدادات EXT_*) يكتب سجل:
    • status = running | completed | failed
    • failureReason — نص سبب الفشل
    • agentKey (مثل "l2.validation"، "ext_esg")
    • agentLayer — الطبقة (L1–L8)
    • correlationId — ربط تشغيل متعدد الوكيل معاً عبر خط أنابيب واحد
    • inputRefs، outputRefs — مراجع للبيانات المستهلكة والمُنتجة
    • confidenceOverall، escalationLevel — ثقة الوكيل والتصعيد

معرفة فشل سير العمل

  • مراقب GET /jobs ويلاحظ failureRatePct وأي وظائف بحالة failed
  • بحث في job.error (من IngestionRun.summary.error) للتفاصيل
  • للعمليات المتقدمة: الاستعلام عن AgentExecution بـ status='failed' و correlationId المطابق
  • كل عملية تحفظ startedAt و finishedAt للحساب الفعلي للمدة والانتظار
  • في الإنتاج، ابحث عن سجلات Pino بـ x-request-id وفلترها حسب "level":"error" لاستخراج تتبع الأخطاء الكامل

النسخ الاحتياطي والتعافي (Backup & Recovery)

الموقع والتشفير

  • جميع النسخ الاحتياطية تُرفع إلى Cloudflare R2 (أو أي S3 متوافق) تحت مسار {S3_PREFIX}/db-backups/
  • اسم الملف: {DATABASE}-{TIMESTAMP}.sql.gz[.enc] (مثال: sigma_pmo-2026-06-28T14-30-45-123Z.sql.gz.enc)
  • تشفير AES-256-GCM من طرف العميل عند تعيين BACKUP_ENCRYPTION_KEY (32 بايت: 64 حرف سادس عشر أو base64)
  • تخطيط التشفير: [12B IV] + [16B auth tag] + [ciphertext]
  • تشفير اختياري من جانب الخادم: S3_SSE=AES256 لـ AWS S3 (في الإنتاج)

الجدولة والتشغيل

  • جدولة: @Cron الساعة 2 صباحاً يومياً (CronExpression.EVERY_DAY_AT_2AM) في BackupService.scheduled()
  • طلب يدوي: POST /backup/run (يتطلب capability canManagePlatform / super-admin فقط)
  • كلا الطريقتين تستدعي BackupService.runBackup() الذي يرجع BackupResult مع ok/error/key/bytes/tables/rows/durationMs
  • حماية ضد التزامن: this.running flag يمنع النسخة الاحتياطية المتزامنة

عملية الإنشاء

  • الخطوة 1: الإدراج المنطقي — استخدم mysql2/promise (لا حاجة لثنائي mysqldump) للاتصال بـ DB وتفريغ الجداول:
    • استرجع قائمة الجداول من information_schema (الترتيب: TABLE_NAME)
    • اطبع CREATE TABLE DDL + INSERT لكل صف (مع معالجة خاصة: JSON كـ UTF-8 نص، NULL آمنة، BLOB، التواريخ كسلاسل)
    • إجمالي rowCount يُحفظ
    • إخراج SQL = SET FOREIGN_KEY_CHECKS=0; SET NAMES utf8mb4; [CREATE + INSERT] SET FOREIGN_KEY_CHECKS=1;
  • الخطوة 2: الضغط — gzip SQL buffer
  • الخطوة 3: التشفير اختياري — AES-256-GCM إذا تم تعيين BACKUP_ENCRYPTION_KEY
  • الخطوة 4: رفع S3 — PutObjectCommand مع ContentType (application/octet-stream للمشفر، application/gzip للضغط البسيط)

تنظيف الاحتياظيات القديمة

  • بعد رفع النسخة الاحتياطية الجديدة، قم بإدراج جميع الكائنات في db-backups/
  • قم بالفرز حسب LastModified (الأحدث أولاً)
  • احتفظ بعدد BACKUP_RETENTION (الافتراضي: 14، قابل للتغيير عبر ENV)
  • احذف الكائنات القديمة عبر DeleteObjectsCommand

الاسترجاع: scripts/restore-db-from-s3.ts

  • npx ts-node scripts/restore-db-from-s3.ts --list — اعرض جميع النسخ الاحتياطية (الأحدث أولاً)
  • npx ts-node scripts/restore-db-from-s3.ts --latest --yes — استرجع أحدث نسخة إلى DB_DATABASE
  • npx ts-node scripts/restore-db-from-s3.ts --key {S3_KEY} --yes — استرجع كائن محدد
  • npx ts-node scripts/restore-db-from-s3.ts --latest --into {DB_NAME} --yes — استرجع إلى DB اختياري (للتحقق الآمن)
  • مطلوب: --yes لتجنب الحذف العرضي (تدمير)
  • العملية: تنزيل → فك التشفير (إن وجد) + gunzip → قنوات mysql (تحديث 100%)

معالجة الأعطال والاستعادة

  • فشل DB — BackupService.dump() تمسك بـ try/catch وترجع خطأ في BackupResult.error؛ السجلات المدمجة تسجل "Nightly backup FAILED: {error}"
  • فشل التخزين S3 — PutObjectCommand أو DeleteObjectsCommand رمي قد يترك نسخة احتياطية جزئية؛ أعد التشغيل الدوري (cron) أو POST /backup/run سيكمل أو يستبدل
  • فشل الاسترجاع — الكتابة إلى mysql قد تفشل في الامتثال DDL (جداول موجودة) أو القيود؛ رسالة الخطأ تتضمن أول 400 حرف
  • تعطل أثناء الاسترجاع — الجدول الهدف قد يكون في حالة وسيطة؛ استخدم --into {scratch_db} للتحقق أولاً

عمليات الملفات والتخزين

  • تحميل الملفات الموجودة: POST /backup/migrate-storage (super-admin) — يمسح المجلد المحلي storageDir ويرفع جميع الملفات إلى R2 تحت {S3_PREFIX}/ (آمنة: يتخطى الملفات الموجودة)
  • الملفات الجديدة — تُرفع مباشرة إلى R2 عند الإنشاء (StorageService)؛ لا يتم حفظها محلياً
  • كل ملف ومحتوى قاعدة البيانات = نسخة احتياطية كاملة من النظام

اتفاقيات مستوى الخدمة (RPO/RTO)

  • RPO (Recovery Point Objective) ~ 24 ساعة: نسخة احتياطية واحدة يومياً (2 صباحاً) + يمكن تشغيل يدوي
  • RTO (Recovery Time Objective) ~ دقائق: الاسترجاع يقرأ من S3، يفك التشفير (AES fast)، gunzip (streaming)، وينفذ SQL؛ عادة ما يكمل في دقائق لقاعدة البيانات متوسطة الحجم
  • الاحتفاظ = 14 نسخة احتياطية افتراضياً (~ 14 يوم من النسخ اليومية)

المراقبة والتحقق

  • GET /backup — اعرض قائمة النسخ الاحتياطية على R2 مع الحجم والوقت (تصفية حسب الشركة/super-admin)
  • سجلات مدمجة: "Nightly backup ok → {key} ({size} MB)" أو "Nightly backup FAILED: {error}"
  • التحقق: استخدم --into sigma_restore_check --yes ثم استعلم عن الجداول دون المخاطرة بالإنتاج

منع الهلوسة (Hallucination Prevention)

كل اقتباس AI مطلوب أن يحمل علامة [SOURCE: externalId] — يُرجع النظام خطأ إذا كانت المصادر في السياق لا تحتوي على هذا المعرف:

  • Source Catalogue (source.entity.ts): جدول curated في DB يحتوي على مراجع حقيقية فقط (FIDIC Red 2017، PMI PMBOK 7، ISO 31000، RICS NRM1/2/3، AACE 18R-97، وغيرها). كل مصدر له externalId فريد (مثل 'FIDIC-RED-2017') وعائلة (FIDIC، PMI، ISO، BIM، AACE، PRIMAVERA)
  • Domain References (domain-references.ts): قائمة Python-style من 20+ مرجعاً علمياً حقيقياً مُنظمة حسب Domain (quantity-survey، cost، classification، procurement، revenue، feasibility، governance)
  • Claude System Prompt: يُخبر Claude صراحة: "never invent a source; ground every substantive claim in the provided reference library and cite it inline using [SOURCE: id]"
  • Fallback when no Claude key: عندما يكون Claude معطّل، تُرجع AiAnalysisService.analyse() enabled: false + قائمة المراجع المتاحة (بدون سردية مختلقة)
  • Citation Extraction (claude.service.ts line 127): regex /\[SOURCE:\s*([A-Za-z0-9._:-]+)\s*\]/g يحصد جميع الاقتباسات من الرد
  • Policy Addons Grounding: نصائح Consultant (correction-plan §2.6) تُدخل في رسالة المستخدم (ليس في system block) حتى تبقى Persona cacheable عبر المشاريع

ربط المراجع والتتبع (Reference Linking & Traceability)

كل قيمة في النظام قابلة للتتبع إلى مصدرها الأولي:

  • ingestionRunId + sourceFileId: كل Activity، Resource، Assignment، Report محفوظة مع معرّفات الدفعة ومسار الملف الأصلي — موجود في AlertDraft.ingestionRunId و AlertDraft.sourceFileId
  • SourceFile entity: مسجل (filename، parser، contentSha256، byteSize، storedPath) — تاريخ كامل من الملفات المرفوعة
  • ConfidenceScore: مؤشرات الثقة مرتبطة بـ ingestionRunId مع breakdown دقيق (perEntity، validation errors/warnings، sourceType، weights)
  • Alert & RuleEvaluation: كل تنبيه يربط rule code + severity + context (الأرقام التي أطلقته) + projectId + activityId (+ resourceId / assignmentId / reportId) + ingestionRunId + sourceFileId — تاريخ كامل من التقييمات
  • RuleEvaluation.summary: JSON مع byCode (عدد التنبيهات per rule)، bySeverity (عدد التنبيهات per severity)، config المستخدمة
  • Persona versioning: كل استدعاء Claude يُسجل مع slug + version + model + stopReason (metadata.user_id = "persona:businessKey:vN[:sim]") — إعادة بناء دقيقة للسلوك

درجات الثقة والموثوقية

ثلاث نقاط قياس متوالية:

  1. Data Confidence (0–1): بعد كل ingestion run، تحسب ConfidenceService درجة شاملة = 0.4×completeness + 0.4×consistency + 0.2×sourceReliability
    • completeness: نسبة الحقول المملوءة / الحقول المطلوبة (project name، activity dates، cost، pct complete، إلخ)
    • consistency: 1.0 − (validation errors×0.1 + warnings×0.02)، مقيدة بـ [0,1]
    • sourceReliability: وزن SourceType (P6 API/XML: 1.0 | MSPROJECT/XER: 0.95 | Excel: 0.85 | CSV: 0.7 | P6 PDF: 0.75)
  2. Council Confidence (0–100%): عندما يُشغّل AiAnalysisService Council mode، يُعيد council.confidence = ثقة الكرسي في الحُكم + agreement = نسبة الأعضاء الذين وافقوا على الموقف الأكثري
  3. Narrative Disclaimer: كل سردية AI مع تحذير صريح: "Advisory analysis grounded in deterministic figures and real scientific sources — for human review, not a final decision" (بالعربية في ar mode)

الفصل AI ← Deterministic

الحد الفاصل واضح:

  • AI computes: السرديات، التحليلات التفسيرية، البدائل المقترحة، ملخصات Executive، نصائح تفصيلية — يستقبل AI أرقاماً حتمية مُحسوبة ويُفسرها
  • Deterministic computes: جميع الأرقام (EVM، SPI، CPI، CPI، EAC، risks، confidence scores، rule evaluations، delay drivers، float، escalation levels)
  • Hybrid: governance decisions = قاعدة معادلات بحتة (لا AI) تُطبق سياسة على تنبيهات لتُقرر من يفعل ماذا ومتى (party، level، notification)؛ ثم AI قد يُصيغ narrative من التنبيهات + القرارات
