# تقرير: التخزين السحابي (S3/R2) والنسخ الاحتياطي التلقائي — Sigma PMO

**التاريخ:** 2026-06-25  ·  **البيئة:** الإنتاج (live) — system.sigma-pmo.com
**مزوّد التخزين:** Cloudflare R2 (متوافق مع S3)  ·  **الحساب:** `1859a4c2…`  ·  **الـBucket:** `sigma-pmo`  ·  **فولدر النظام:** `sigma-system/`

---

## 1) الهدف
تفعيل تخزين سحابي حقيقي لملفات المنصّة عشان نقل الملفات والتطبيق يبقى أسرع وأخف على السيرفر، مع
**فولدر باسم النظام** تتجمّع فيه كل ملفات سيجما، و**نسخ احتياطي تلقائي كامل** لكل البيانات + الملفات.

## 2) اللي اتعمل (ملخّص تنفيذي)

| # | البند | الوصف | الحالة |
|---|-------|--------|--------|
| 1 | تخزين الملفات على R2 | كل ملف بيترفع بيتخزّن على R2 تحت فولدر `sigma-system/` (بمفتاح SHA-256 — بدون تكرار). | ✅ |
| 2 | فولدر النظام (`S3_PREFIX`) | إضافة بادئة قابلة للضبط تخلّي كل الملفات + النُسخ تحت فولدر واحد باسم النظام. | ✅ مدمج |
| 3 | نسخ احتياطي تلقائي للداتا | مهمة ليلية (2 ص) **داخل التطبيق** تعمل نسخة كاملة لقاعدة البيانات → R2، مشفّرة AES-256، مع الاحتفاظ بآخر 14 نسخة. | ✅ مدمج |
| 4 | نسخ يدوي عند الطلب | السوبر-أدمن يقدر يشغّل نسخة فورًا ويشوف النُسخ من الداشبورد (`POST /backup/run` · `GET /backup`). | ✅ |
| 5 | مزامنة الملفات القديمة | أداة (وendpoint) ترفع أي ملفات قديمة كانت على ديسك السيرفر إلى R2 — بحيث **الملفات + الداتا كلها** على السحابة. | ✅ |
| 6 | بدون اعتماد خارجي | النسخ بيتعمل بـ`mysql2` مباشرة (من غير الحاجة لأداة `mysqldump`) فيشتغل جوّه الحاوية ومع MySQL 8.4. | ✅ |

> **تفاصيل تقنية:** التغييرات اترفعت على `main` (الإنتاج) — موديول `BackupModule` + بادئة `S3_PREFIX` في `StorageService`،
> وسكربتات `db:backup` / `db:restore` / `storage:migrate` / `s3:check`. النُسخ والملفات تحت نفس الفولدر `sigma-system/`.

## 3) الاختبار الفعلي على الإنتاج (النتائج)

اتعمل اختبار حقيقي على بيانات وملفات الإنتاج الحقيقية، والنتيجة ظهرت فعليًا في R2:

**`s3://sigma-pmo/sigma-system/` → 72 عنصر · 59.59 ميجابايت**

| العنصر | النتيجة |
|--------|---------|
| ملفات سيجما المنقولة إلى R2 | **71 ملف حقيقي** (تقارير يومية · Baseline بريمافيرا XER · حصر كميات BOQ بصيغة xlsx · مطالبات JSON …) |
| نسخة قاعدة البيانات | **77 جدول · 4,619 صف** — نسخة كاملة |
| سلامة المحتوى العربي | ✅ محفوظ بالكامل (عيّنة متحقَّقة: «مطالبة تمديد المدة») |
| التشفير | ✅ AES-256-GCM قبل الرفع |
| قابلية الاسترجاع | ✅ متحقَّق منها: فك التشفير ← فك الضغط ← مطابقة 1:1 (round-trip) |
| حجم النسخة | ~832 كيلوبايت (مضغوطة ومشفّرة) |

**الخلاصة:** كل الملفات بتتخزّن، والداتا بتتنسخ نسخة كاملة سليمة وقابلة للاسترجاع — وكله ظاهر في R2.

## 4) الأمان
- المفاتيح السرية (R2 + مفتاح تشفير النسخ) **مخزّنة في متغيّرات بيئة Coolify فقط** — مش متكتوبة في الكود ولا جيت إطلاقًا.
- النسخة الاحتياطية **مشفّرة** قبل ما تغادر السيرفر (تحتوي على بيانات كل العملاء).
- مفتاح التشفير (`BACKUP_ENCRYPTION_KEY`) **مطلوب للاسترجاع** — لازم يتحفظ في مكان آمن؛ من غيره مفيش استرجاع.
- يُنصح بعمل **تدوير (rotate)** لمفاتيح R2 بعد التشغيل لأنها اتبعتت كنص.

## 5) الخطوة المتبقية (تبديل البيئة — Cutover)
التطبيق دلوقتي شغّال على R2 لكن على bucket قديم اسمه `hmis-backup`. عشان يستخدم `sigma-pmo` + فولدر النظام،
نغيّر في **resource الباك إند بالإنتاج → Environment Variables** القيم دي ونعمل Redeploy:

```
S3_BUCKET=sigma-pmo
S3_ACCESS_KEY=<مفتاح sigma-pmo الجديد>
S3_SECRET_KEY=<المفتاح السري الجديد>
S3_PREFIX=sigma-system
# تُترك كما هي: S3_ENDPOINT · S3_REGION=auto · S3_FORCE_PATH_STYLE=true
# لا تُغيَّر: BACKUP_ENCRYPTION_KEY (وإلا النُسخ القديمة مش هتسترجع)
```

بعد الـRedeploy: أي ملف جديد يروح `sigma-pmo/sigma-system/`، والنسخة الليلية تروح `sigma-pmo/sigma-system/db-backups/`.
(الملفات القديمة محفوظة على الاتنين، فمفيش أي فقد بيانات.)

## 6) ملاحظات تشغيلية
- **الجدولة:** نسخة تلقائية كل ليلة 2 صباحًا + إمكانية التشغيل اليدوي من الداشبورد.
- **الاسترجاع:** `npm run db:restore -- --latest --yes` (يحتاج نفس مفتاح التشفير).
- **الاحتفاظ:** آخر 14 نسخة (قابلة للتعديل بـ`BACKUP_RETENTION`).
- **اختياري:** تحديث مسارات الملفات القديمة في قاعدة البيانات لتقرأ من `sigma-pmo` (توحيد كامل على bucket واحد) — جاهز عند الطلب.
