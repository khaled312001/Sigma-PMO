1نظرة عامة على المنصة

Sigma PMO هو نظام تشغيل للحوكمة مدعوم بالذكاء الاصطناعي (AI Governance Operating System) — ليس أداة تقارير، بل نظام دعم قرار حوكمي (Governance Decision Support System). كل وظيفة في المنصة عبارة عن وكيل (Agent) مستقل يتبع نموذج تشغيل موحّد من سبعة حقول: الهدف (Objective) · المدخلات (Inputs) · منطق المعالجة (Processing Logic) · المخرجات (Outputs) · درجة الثقة (Confidence Score) · منطق التصعيد (Escalation Logic) · سجل التدقيق (Audit Trail).

تضاف فوقها وكلاء امتداد (Extension Agents) دون أي تعديل هيكلي — وأبرزها وكيل الاستثمار ودراسات الجدوى ext.investment (القسم 5) ووكيل الاستدامة ext.esg.

مبدأ أساسي: «الحتمية أولاً» (Deterministic-first) — كل الأرقام (EVM، NPV، IRR، درجات المخاطر) تُحسب بمعادلات حتمية قابلة للتدقيق، والذكاء الاصطناعي يقتصر دوره على الصياغة والسرد وخلف بوابة اعتماد بشري دائماً.

كما تدعم المنصة هرمية حوكمة كاملة (Enterprise → Portfolio → Program → Project)، ودورة حياة (Initiation → Closure)، وحالة حوكمة رباعية: أخضر · أصفر · برتقالي · أحمر.

2تسجيل الدخول والحسابات

صفحة تسجيل الدخول /auth

الهدفبوابة الدخول الموحّدة. صُممت بشكل عصري متحرك (Aurora animation + طبقات L0→L8) وغير مرتبطة بمشروع بعينه.
طريقة الاستخدام
  1. اختر نوع المستخدم (الدور) من شريط الأدوار — سيُملأ البريد وكلمة المرور تلقائياً للحسابات التجريبية.
  2. أو اكتب البريد وكلمة المرور يدوياً ثم اضغط Sign in.
  3. عند النجاح يصدر النظام مفتاح API جديداً ويبطل القديم تلقائياً (حماية للجلسات القديمة)، ثم يحوّلك للوحة الرئيسية.
ضمن المنظومةكل طلب لاحق يحمل المفتاح في ترويسة x-api-key، والباك إند يفرض صلاحيات الدور على كل نقطة وصول (الواجهة تخفي الأزرار فقط؛ الفرض الحقيقي في الخادم).

3عناصر الواجهة المشتركة

القائمة الجانبية (Sidebar)

منظمة حسب طبقات الحوكمة: GOVERNANCE COMMAND (الوجهات الاستراتيجية) ثم AGENT LAYERS · L0–L8 (كل طبقة بوسمها) ثم TOOLS & EVIDENCE ثم ADMIN. قابلة للطي عبر الزر الدائري عند الحافة (تتحول لشريط أيقونات 68px وتُحفظ حالتها)، ومتجاوبة مع الموبايل (قائمة منزلقة). كل دور يرى قائمته الخاصة فقط حسب صلاحياته.

تبديل المشروع (Project Switcher)

طريقة الاستخداماضغط على شارة المشروع في الشريط العلوي لفتح القائمة، واختر المشروع. كل الصفحات تعيد تحميل بياناتها فوراً للمشروع المختار (التنبيهات، الملخصات، التحليلات، المخاطر…) ويُحفظ اختيارك للجلسات القادمة.

الوضع الداكن والفاتح (Dark / Light)

زر الشمس/القمر في الشريط العلوي يبدّل الثيم فوراً ويحفظ التفضيل. كل المخططات والشارات تتكيف تلقائياً.

اللغة العربية (RTL)

زر اللغة (ع / E) يبدّل بين الإنجليزية والعربية مع انعكاس كامل لاتجاه الواجهة (RTL) — القوائم والجداول والمخططات.

4قيادة الحوكمة (Governance Command)

اللوحة الرئيسية — Overview /

الهدفنبضة سريعة للمشروع المختار: عدد عمليات الاستيراد، إجمالي التنبيهات (الحرجة/التحذيرية)، ثقة آخر استيراد، توزيع التنبيهات حسب القاعدة، وآخر ملخص تنفيذي أسبوعي.
طريقة الاستخدامالبطاقات الأربع العلوية روابط مباشرة (Input · Review · Approval · Evidence). بدّل المشروع من الأعلى لترى نبضة المشروع الآخر فوراً.
من يراهاجميع الأدوار (كلٌ ببيانات نطاقه).

مركز القيادة — Command Center (L8) /governance-command

الهدفالواجهة المرئية لوكيل L8 (السلطة النهائية): يجمع مخرجات كل الوكلاء لكل عقدة (مشروع/برنامج/محفظة/منشأة) في حالة واحدة معتمدة (أخضر/أصفر/برتقالي/أحمر) مع درجة رقمية مفسَّرة، ويولّد إجراءات تصحيحية قابلة للتتبع.
طريقة الاستخدام
  1. راقب توزيع الحالات والشجرة الهرمية مع شارة حالة لكل عقدة.
  2. اضغط Recompute لإعادة التجميع بعد أي بيانات جديدة (إعادة الحساب آمنة التكرار idempotent).
  3. اعمل على قائمة Corrective actions: غيّر حالتها (open → in-progress → done) مع المالك.
  4. انزل لتفاصيل أي عقدة لرؤية سجل تنفيذ كل وكيل (الدليل الكامل لكل رقم).
من يراهاطبقة الحوكمة (Admin · Reviewer · Client · Consultant · Contractor) — لا تظهر لمقاول الباطن.

لوحة الإدارة التنفيذية — Executive (L7) /executive

الهدفرؤية مجلس الإدارة: حالة الحوكمة، صحة الجدول، صحة التكلفة، نسبة التجاوز المتوقعة، التعرض للمخاطر، والنتائج الحرجة — في سطر عنوان واحد وبطاقات KPI.
طريقة الاستخدامتُقرأ من أعلى لأسفل: العنوان التنفيذي ← بطاقات KPI ← التفاصيل. بدّل المشروع لمقارنة المحفظة. كل رقم له أصل حتمي يمكن تتبعه من سجل الوكلاء.

هيكل الحوكمة — Hierarchy /hierarchy

الهدفإدارة الهرمية الرباعية وربط المشاريع بالبرامج والمحافظ، مع تجميع الحالة «الأسوأ يطفو لأعلى» (worst-of-children roll-up).
طريقة الاستخدامأنشئ العقد بالأزرار العلوية، ثم اربط المشروع بعقدة عبر Attach، وحدّد مرحلة دورة الحياة لكل عقدة. زر Recompute يعيد تجميع الحالات صعوداً.
من يراهاالإدارة عبر صلاحية canManageHierarchy (Admin والعميل)، والاطلاع لطبقة الحوكمة.

سجل الوكلاء — Agent Registry /agents

الهدفإثبات «العقد الموحّد»: كل الوكلاء (L0–L8 + الامتدادات) يظهرون تلقائياً بنفس البطاقة، مع سجل التنفيذات (AgentExecution) لكل تشغيل: المُدخلات، الثقة، التصعيد، الحالة.
طريقة الاستخدامشغّل أي وكيل يدوياً بزر Run (أو الخط الكامل L1→L8 بزر Run pipeline)، وراجع تاريخ التنفيذات أسفل الصفحة. أي وكيل جديد يُضاف للمنصة يظهر هنا تلقائياً بدون أي تعديل.

المشاريع — Projects /projects

الهدفكل المشاريع الحالية في النظام (تنشأ تلقائياً من الاستيراد في L1) مع بياناتها الأساسية وتاريخ البيانات (Data Date)، ومنها تستطيع تعيين المشروع الحالي.

5الاستثمار ودراسات الجدوى — Investment & Feasibility Intelligence

قدرة متكاملة تدعم المستثمر من مرحلة الفكرة حتى جاهزية التمويل، على مستويين + مسار الإسكتشات، ومسجلة كوكيل امتداد ext.investment بحيث يُدقَّق كل تقييم في سجل الوكلاء.

قائمة الفرص — المستوى الأول /feasibility

الهدفالتقييم الاستثماري السريع (Rapid Investment Assessment): بمدخلات قليلة فقط (نوع المشروع، الموقع، حجم الاستثمار، هيكل التمويل، الهدف) يولّد النظام تقييماً مبدئياً كاملاً اعتماداً على مكتبة افتراضات مرجعية مُدارة بإصدارات.
طريقة الاستخدام
  1. اضغط New opportunity واملأ النموذج (يكفي نوع المشروع + المدينة + حجم الاستثمار أو مساحة البناء BUA).
  2. اضبط هيكل التمويل: نسبة حقوق الملكية / الدين، الفائدة، مدة السداد.
  3. افتح الفرصة وشغّل Run rapid assessment.

نتائج التقييم — NPV · IRR · Payback · DSCR

الهدفالنتائج تشمل: الجدوى المبدئية، تفصيل CAPEX، تقديرات OPEX، توقعات الإيرادات مع منحنى التشغيل التدريجي، NPV مبدئي، IRR للمشروع وللملكية، فترة الاسترداد، DSCR (تغطية خدمة الدين)، تصنيف مخاطر مُسمّى الأسباب، وتوصية حوكمية: Proceed · Proceed with Conditions · Hold · Reject — مرتبطة بالحالة الرباعية (أخضر/أصفر/برتقالي/أحمر).
ضمن المنظومةكل تشغيل يُحفظ append-only مع لقطة المدخلات والافتراضات المستخدمة (قابلية إعادة الإنتاج للأبد)، ويسجَّل كتنفيذ مدقَّق للوكيل ext.investment بدرجة ثقة مبنية على اكتمال المدخلات. كل شرط في التوصية مُسمّى صراحة («رفع IRR إلى عتبة 12%»…).

الدراسة الاحترافية — المستوى الثاني (Bankability Engine)

الهدفإذا كانت الفرصة واعدة، يولّد زر Generate professional study دراسة كاملة من 17 قسماً: الملخص التنفيذي، تقييم السوق، الجدوى الفنية، النموذج التشغيلي، تحليل CAPEX & OPEX، توقعات الإيرادات، التدفقات النقدية، القوائم المالية (بجدول إطفاء حقيقي للدين وضريبة 9%)، تحليلات NPV وIRR وفترة الاسترداد وDSCR، تحليل الحساسية (6 سيناريوهات معاد حسابها بالكامل)، تقييم المخاطر، متطلبات التمويل، تقييم القابلية للتمويل البنكي (6 معايير نجاح/فشل)، والتوصية الحوكمية.
طريقة الاستخدامافتح أي قسم للاطلاع، ثم Approve section — الاعتماد البشري هو البوابة قبل التضمين النهائي في الباقات. إعادة التوليد تنشئ نسخة جديدة وتحفظ القديمة (versioned).

باقات الجمهور — Investor · Partner · Bank

الهدفنفس الدراسة تُركَّب تلقائياً في ثلاث باقات موجهة: Investor Package (قصة العائد)، Partner Package (قصة المشروع)، Bank Financing Package (قصة الائتمان: قوائم مالية + DSCR + متطلبات تمويل). زر Print / save PDF يخرج الباقة مستندَ تسليم.

الإسكتشات والرسومات الأولية (Concept Sketches)

الهدفكثير من المستثمرين يبدأون بإسكتش يدوي أو ملاحظات مكتوبة. ترفع الملف (صورة أو PDF حتى 15MB) فيستخلص الذكاء الاصطناعي (OCR + رؤية حاسوبية) البيانات المتاحة: مساحة الأرض، مساحة البناء BUA، عدد الأدوار، المناطق الوظيفية، الأبعاد التقريبية، مزيج الوحدات، السعة، الملاحظات المكتوبة، والافتراضات.
طريقة الاستخدام
  1. ارفع الملف من تبويب Concept sketches.
  2. اضغط AI extract — يقترح النظام الحقول (اقتراح فقط، لا يُطبَّق شيء).
  3. راجع وعدّل في نموذج Review & confirm ثم اعتمد — أنت بوابة الاعتماد؛ عندها فقط تُدمج الحقول في مدخلات الجدوى.
  4. أعد تشغيل التقييم: إن لم يوجد رقم استثمار، يشتق النظام CAPEX من BUA × benchmark تكلفة/م² × معامل الموقع.
بدون مفتاح Anthropic مُهيأ، يتحول المستند لوضع الإدخال اليدوي بنفس نموذج الاعتماد — لا يتعطل المسار.
من يراهاAdmin · Reviewer · Client · Consultant (صلاحية canRunFeasibility). بيانات الاستثمار محجوبة عن المقاول ومقاول الباطن.

6طبقات الوكلاء L0 → L8 صفحةً صفحة

L0 · المعرفة والقواعد /knowledge

الهدفالأساس المعرفي لكل الوكلاء: مكتبة قواعد Sigma (عتبات الانحراف، قواعد الموارد…)، سجل المصادر (FIDIC · PMI · ISO)، الأطر والسياسات، والدروس المستفادة القابلة للنمو.
طريقة الاستخدامتنقّل بين التبويبات الأربعة. أي اقتباس ذكاء اصطناعي في المنصة يستشهد بمعرّفات هذه المصادر بصيغة [SOURCE: id].

L1 · إدخال البيانات /input

الهدفبوابة دخول البيانات: جداول Primavera/MS Project، تقارير التقدم، وسجلات المشروع (RFI/Submittal/NCR/تغييرات/مشتريات/موارد/تكلفة/صور موقع).
طريقة الاستخداماسحب الملف أو اختره، فيُحفظ نسخة أصلية مؤرشفة (content-addressed)، ويُحلَّل ويُطبَّع في النموذج القانوني بدون استبدال (إصدارات append-only)، وتحصل كل عملية على درجة ثقة. راقب الجدول السفلي لحالة كل عملية.
من يراهاأصحاب صلاحيات الرفع (Admin، المقاول لجداوله وتقاريره، مقاول الباطن لتقارير أنشطته فقط).

L2 · المراجعة /review

الهدفواجهة محرك القواعد: زر Evaluate يفحص بيانات المشروع الحالي ضد مكتبة القواعد ويولّد تنبيهات (حرج/تحذير/معلومة)، ثم يصدر قرارات الحوكمة المرتبطة تلقائياً.
طريقة الاستخدامشغّل التقييم، رشّح بالخطورة، وافتح أي تنبيه لرؤية قراره. زر Weekly summary يولّد الملخص التنفيذي (حتمي أو بصياغة AI إذا فُعّل) للمشروع المختار.

L3 · أرشيف القرارات /decisions

الهدفسجل قرارات L3 الكامل للمشروع المختار: الطرف المسؤول، بند FIDIC المنطبق ومهله، مستوى التصعيد (L1–L3)، وما إذا اعتُمد القرار أو رُفض أو جرى الإقرار به.

L4 · التحليلات والقيمة المكتسبة /analytics

الهدفحساب القيمة المكتسبة بصيغ PMI الحتمية من بيانات الأنشطة والتقارير: PV · EV · AC ← SPI · CPI · EAC · ETC · VAC، مع مخططات الاتجاه وتوقع الانحراف.
طريقة الاستخداماختر المشروع وشغّل التحليل؛ كل لقطة KPI تُحفظ بتاريخ بياناتها لتتبع الاتجاه عبر الزمن.

L5 · سجل المخاطر /risk

الهدفتوليد وتقييم المخاطر حتمياً (احتمال × أثر) من إشارات المشروع (انزلاقات، تجاوزات تكلفة، NCRs…) مع أولويات معالجة وخريطة حرارية.
طريقة الاستخدامزر التشغيل يستدعي وكيل l5.risk فيحدّث السجل؛ المخاطر عالية الدرجة تغذّي التصعيد والحالة الرباعية تلقائياً.

L6 · المطالبات والنزاعات /claims

الهدفتحليل تأخير حتمي (مخطَّط مقابل فعلي، نافذةً نافذة)، تحديد المطالبات المحتملة، إسناد المسؤولية، وربط كل مطالبة بأدلتها — تمهيداً لخطابات FIDIC في صفحة Letters.

طبقتا L7 وL8 شُرحتا في القسم 4 (Executive وCommand Center) — هما نفس الصفحتين ضمن تبويب طبقات الوكلاء.

7الأدوات والأدلة (Tools & Evidence)

مستودع الوثائق /repository

الهدفكل سجلات L1 للمشروع (RFIs · Submittals · NCRs · أوامر التغيير · المشتريات · الموارد · التكلفة · صور الموقع) في جدول واحد قابل للترشيح، مع إضافة سجلات جديدة يدوياً.

الأدلة /evidence

الهدفالإجابة الدائمة على سؤال «من أين جاء هذا؟»: لكل تنبيه حزمة دليل تضم الصف الخام الأصلي، الملف المصدر وبصمته SHA-256، درجة الثقة، والمنطق الذي أطلق القاعدة.
طريقة الاستخداماختر تنبيهاً من القائمة (مرشّحة بالمشروع الحالي) لعرض حزمته كاملة.

الاعتماد /approval

الهدفبوابة الاعتماد البشري لقرارات L3: كل قرار يظهر مع تنبيهه ومسؤوليته وبند FIDIC، وتُسجَّل كل مراجعة بهوية مراجعها (سجل تدقيق دائم).

خطوط الأساس /baselines · المحاكاة /simulation

الهدفBaselines: إدارة خطوط الأساس الزمنية ومقارناتها. Simulation: تفريع سيناريو رملي (Sandbox) تجرب فيه «ماذا لو» (تأخير نشاط، تغيير موارد) بأمان كامل — الكتابة في فرع السيناريو فقط، ثم تُرقّى النتيجة لقاعدة الحقيقة بقرار صريح أو تُهمل.

التعارضات /clashes · المخططات /drawings

الهدفClashes: استيراد تقارير التعارض ومتابعة الحل والمسؤولية. Drawings: رفع حزم المخططات؛ يستخرج النظام بياناتها (كعدد الأدوار) لتغذية مولّد خط الأساس.

الخطابات /letters · سجل المصادر /sources

الهدفLetters: صياغة خطابات FIDIC (إشعارات، ردود) بمساعدة الذكاء الاصطناعي مع استشهاد إلزامي بالمصادر واعتماد بشري قبل أي اعتماد نهائي — لا إرسال آلي إطلاقاً. Sources: السجل المرجعي (FIDIC Red Book، PMI، ISO…) الذي تُسند إليه الاقتباسات.

التقرير الشهري /reports/monthly · AI مقابل الإنسان /comparison

الهدفMonthly reports: توليد التقرير الشهري (عربي/إنجليزي) من بيانات المشروع الحقيقية مع اعتماده وتصديره. AI vs Human: رفع مخرج بشري ومقارنته بمخرج المنصة وتسجيل حكم بشري — تغذية راجعة لتحسين الجودة.

8صفحات الإدارة (Admin)

صلاحيات الأدوار /admin/roles

الهدفتحكم الأدمن الفعلي في صلاحيات بقية الأدوار أثناء التشغيل: أي مفتاح يُقلب هنا يسري فوراً على الخادم (الطلب التالي للمستخدم المتأثر يُرفض أو يُقبل) وعلى واجهته (تختفي/تظهر الصفحات).
طريقة الاستخدامبدّل أي خلية في المصفوفة. عمود الأدمن مقفول (لا يمكن إقفال الأدمن خارجاً)، وصلاحيتا canRead وcanManageRoles محميتان. Reset يعيد الافتراضات لدور أو للكل.

السياسة /admin/policy · الشخصيات /admin/personas

الهدفPolicy: تهيئة قواعد القرار (عتبات الخطورة، مستويات التصعيد، الأطراف المُخطرة) بنسخ متتابعة لا تُستبدل. Personas: إدارة شخصيات الخبراء (مخطِّط، خبير FIDIC، محلل مخاطر…) التي يتكلم النظام من خلالها — append-only بالكامل.

المستخدمون /admin/users

الهدفدورة حياة المستخدمين كاملة: إنشاء مستخدم بدور ونطاق مشاريع، تعديل (الاسم/الدور/التفعيل)، إعادة تعيين كلمة مرور، تدوير مفتاح API (يظهر مرة واحدة)، وحذف — مع حماية آخر أدمن نشط من التعطيل أو الحذف.

الإعدادات المتقدمة /admin/settings · سجل التدقيق /audit

الهدفSettings: مفتاح الذكاء الاصطناعي يُحفظ مشفّراً في قاعدة البيانات ويسري فوراً. Audit: تتبع زمني للعمليات (استيراد، قرارات، مراجعات…) لأغراض الحوكمة والتسليم.

9الأدوار الستة — ماذا يرى كل مستخدم

لكل دور قائمة جانبية وصلاحيات مختلفة فعلياً — ليس إخفاءً شكلياً: الخادم يرفض أي طلب خارج صلاحية الدور حتى لو طُلب مباشرة.

Sigma Admin — مدير المنصة

يرى كل شيء ويدير كل شيء (الأقسام 1–8 أعلاه كلها بصلاحياته). حصرياً: مصفوفة الصلاحيات، المستخدمون، الشخصيات، الإعدادات المتقدمة، وComputer Use.

Sigma Reviewer — المراجع الداخلي

ميثاق الدورعين Sigma الفاحصة: اطلاع كامل على كل المشاريع، تشغيل التقييمات والملخصات والمقارنات (AI vs Human)، بدون رفع بيانات أو تعديل سياسة أو محاكاة أو إدارة مستخدمين. يملك صلاحية دراسات الجدوى.

Client — العميل (مالك القرار)

ميثاق الدورصاحب القرار: اطلاع كامل، تعديل السياسة، اعتماد الخطابات وخطوط الأساس والقرارات، إدارة الهرمية، المحاكاة، ودراسات الجدوى (هو المستثمر غالباً). لا يرفع بيانات تشغيلية ولا يدير المستخدمين.

Consultant — الاستشاري

ميثاق الدورقراءة + اقتراح + محاكاة: اطلاع كامل وتقييم القواعد والتحليلات والملخصات والمحاكاة ودراسات الجدوى — بدون رفع بيانات أو اعتمادات أو تعديل سياسة.

Contractor — المقاول

ميثاق الدورطرف التنفيذ: يرفع الجداول والتقارير وBoQ والخطابات، يقيّم قواعد شريحته ويولّد ملخصاتها، ويحاكي في فرع رملي. لا يرى المستويات الأعلى من الهرمية (Enterprise/Portfolio) ولا بيانات الاستثمار، ولا يملك أي اعتماد.

Subcontractor — مقاول الباطن

ميثاق الدورأضيق نطاق: يرفع تقارير التقدم لأنشطته المسندة فقط (activityScope) ويحاكي عليها، ولا يرى الوضع المالي العام للمشروع ولا صفحات الحوكمة — وأي مسار خارج نطاقه يقابله حاجز الصلاحيات أعلاه.
